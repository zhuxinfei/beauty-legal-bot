# 美妆法务资讯雷达优化设计

## 1. 背景与目标

当前 `beauty-legal-bot` 已实现 Cloudflare Worker 定时触发、DeepSeek 生成、飞书推送、KV 去重的基础能力。但它主要依赖 prompt 中的概括性信息源，无法系统利用用户提供的 Excel 信息源，也无法稳定生成带原文链接的完整资讯页面。

本次优化目标是把项目升级为“美妆法务资讯雷达”：

- 每周一北京时间 08:00 自动运行，不依赖本地设备。
- 使用 DeepSeek API 作为唯一模型引擎，但代码层保留可替换引擎边界。
- 利用 Excel 中的官网、媒体、公众号等信息源，形成结构化源目录。
- 先广泛采集候选信息，再严格筛选，宁缺毋滥。
- 新法规、新规修订、废止、生效提醒等内容优先推送，相关则尽量全收。
- 每个模块分别选取高价值内容，不是总共只推 3-6 条。
- 飞书只推摘要卡片，并附完整资讯页面链接。
- 完整资讯页面由 Cloudflare Worker 生成，海内外均可访问。

## 2. 参考调研结论

调研过的相关方案包括：

- Horizon / TrendRadar：开源资讯雷达，适合参考“多源采集 → LLM 筛选 → 多渠道推送”的管道设计。
- changedetection.io：适合参考网页变更监控与摘要，但完整引入过重。
- RSSHub / RSS-Bridge：适合在部分源可用时作为适配器参考，但不能作为强依赖。
- 商业监管情报平台：RegPulse、Polzia、RegGuard 等说明“风险评分、截止日、影响范围、行动建议”是法务情报产品的关键结构。

结论：不直接引入大型系统，采用轻量、专用、可部署在 Cloudflare Worker 的架构。

## 3. 信息源设计

### 3.1 源目录

Excel 中的信息源会转换成 `worker/sources.json`。表格里多数“分类”为空，因此转换时会先按站点名称、URL、关键词做自动补全，再把结果保存成可人工修正的配置文件。后续维护只改 `sources.json`，不改主程序。

每个源包含：

```json
{
  "name": "泰国 FDA",
  "url": "https://www.fda.moph.go.th/",
  "module": "新规及案例动态",
  "region": "亚洲",
  "country": "泰国",
  "source_type": "official_site",
  "authority_type": "regulator",
  "priority": "high",
  "topics": ["化妆品", "禁限用物质", "备案", "进口"]
}
```

### 3.2 源分类

信息源分为三层：

1. **官方可抓取源**
   - NMPA、SAMR、各地药监局/市监局、海关、FDA、EU、BPOM、Thai FDA、AICIS、EPA 等。
   - 用于获取可验证的标题、日期、原文链接。

2. **公开行业网站**
   - 行业媒体、合规服务网站、协会网站、知识产权机构。
   - 用于补充法规解读、案例线索和行业动态。

3. **公众号源**
   - 不直接爬微信全文。
   - 作为 DeepSeek 检索线索和选题雷达。
   - 如能找到公开网页或官方原文，则输出原文链接；否则只作为参考线索，不编造链接。

### 3.3 地区与国家粒度

最终报告以“模块”为主线，但每条内容必须标注国家/地区：

- 亚洲：中国、印尼、泰国、越南、菲律宾、马来西亚、日本、韩国等。
- 欧洲：欧盟、法国、德国、意大利、西班牙、英国等。
- 北美：美国、加拿大、墨西哥。
- 大洋洲：澳大利亚、新西兰。
- 其他地区后续可通过 `sources.json` 扩展。

## 4. 时间窗口与筛选原则

### 4.1 时间窗口

法务资讯不按普通新闻周期处理，采用以下窗口：

- **法规动态默认窗口：过去 14 天**。
- **法规关键节点窗口：未来 90 天**，包括生效日、征求意见截止、整改截止、强制认证节点。
- **案例窗口：过去 30 天**。
- **高代表性案例窗口：最多 60 天**，但必须解释为什么仍值得关注。
- **行业媒体/公众号线索：过去 14 天**。

### 4.2 宁缺毋滥

- 不为了覆盖区域或模块硬凑。
- 某模块没有高价值内容时显示“本周无高价值更新”。
- 公众号、行业媒体信息必须回到官方源或公开网页佐证；无法佐证时降低优先级。
- 所有内容必须有明确法务价值：准入、备案、标签、广告、进出口、IP、处罚、诉讼、召回、认证、平台监管。

### 4.3 新法规优先规则

只要是美妆行业直接或间接相关的以下内容，相关且真实则尽量全推：

- 新法规发布。
- 法规修订。
- 法规废止。
- 征求意见。
- 正式生效。
- 生效提醒。
- 禁限用物质清单变化。
- 认证、备案、注册、标签、进出口要求变化。

## 5. 输出结构

### 5.1 飞书摘要卡片

飞书只推摘要，控制在 300-600 字：

```md
⚖️ 美妆法务周报

本周新增 4 条法规动态、3 条案例、2 条进出口提示。

重点：
- [法规][亚洲-印尼] BPOM 清真认证进入关键节点
- [案例][亚洲-中国] 直播功效宣称处罚持续高压
- [法规][欧洲-欧盟] SCCS 调整防晒剂安全意见

风险提醒：
- 未来 90 天内有 2 个法规节点
- 印尼/泰国市场需优先排查备案与标签

查看完整周报：[打开页面](url)
```

### 5.2 完整资讯页面

完整页面由 Cloudflare Worker 生成 HTML，路径：

```text
/report/latest
/report/2026-05-25
```

页面模块：

1. 新规 / 修订 / 废止 / 生效提醒
2. 广告合规及处罚案例
3. 美妆行业动态
4. 知识产权动态
5. 进出口 / 跨境电商动态

每个模块选取 2-3 条高价值内容；“新规/修订/废止/生效提醒”模块相关则尽量全推。

### 5.3 单条资讯结构

法规类：

```md
[法规][亚洲-泰国] Thai FDA 更新化妆品禁限用清单
- 原文：[Thai FDA](url)
- 状态：正式发布 / 征求意见 / 生效提醒 / 废止
- 时间：发布日期；生效日/反馈截止日
- 内容：核心条款，分点换行
- 影响范围：涉及品类、渠道、市场、企业类型
- 行动建议：法务/注册/供应链/电商团队该做什么
```

案例类：

```md
[案例][亚洲-中国] 某美妆品牌直播虚假宣传被罚
- 原文：[上海市监局](url)
- 案情：谁、在哪里、做了什么
- 处罚/判决：金额、措施、法律定性
- 拆解分析：为什么被罚，执法逻辑是什么
- 合规启示：我们的直播、电商详情页、达人合作是否有同类风险
```

行业动态类：

```md
[动态][欧洲-法国] 某协会发布美妆环保宣称指南
- 原文：[来源](url)
- 内容：发生了什么
- 合规意义：为什么值得法务关注
- 建议：是否需要更新内部规则
```

知识产权类：

```md
[IP][北美-美国] 某品牌包装外观诉讼进展
- 原文：[来源](url)
- 案情：争议焦点
- 结果/进展：判决或程序状态
- 启示：对包装、命名、营销素材的影响
```

进出口类：

```md
[进出口][亚洲-印尼] BPOM 更新进口化妆品备案要求
- 原文：[BPOM](url)
- 内容：政策变化
- 影响范围：进口商、跨境平台、SKU 类型
- 行动建议：资料准备、备案、标签、清关注意事项
```

## 6. 页面 UI/UX 设计

页面采用专业法务情报报告风格：

- 背景：浅灰白 `#F8FAFC`。
- 主色：深蓝灰 `#1E3A5F`。
- 风险强调：琥珀色 `#D97706`。
- 正文：深灰 `#0F172A`。
- 卡片：白底、轻边框、轻阴影。
- 字体：系统中文字体优先，保证国内外访问稳定。
- 正文行高 1.6 左右，移动端最小字号 16px。
- 所有原文链接是明显按钮或文本链接，不显示裸 URL。

页面结构：

1. 顶部：标题、周期、更新时间、风险标签。
2. 执行摘要：3-5 条本周最重要风险。
3. 风险雷达：高/中/低风险数量，涉及国家标签。
4. 模块内容：各模块卡片。
5. 底部：生成时间、源说明、AI 生成免责声明。

移动端：

- 单列卡片。
- 区域/国家标签可换行展示。
- 原文链接按钮触控面积不低于 44px。
- 避免复杂交互，保证飞书内置浏览器可读。

## 7. 技术架构

### 7.1 文件结构

```text
worker/
  index.js              # Worker 主逻辑
  sources.json          # 从 Excel 转换出的信息源目录
  wrangler.toml         # Cron、KV、变量配置

beauty_legal_bot.py     # 本地测试脚本，可同步核心 prompt/渲染逻辑
README.md               # 部署和使用说明
docs/superpowers/specs/ # 设计规格文档
```

### 7.2 数据流

```text
sources.json
   ↓
Cloudflare Cron（每周一 08:00 北京时间）
   ↓
fetch sources
  - HTML 列表抓取
  - RSS/RSSHub 抓取（可用则用）
  - 公众号作为 prompt 线索
   ↓
normalize candidates
  title/url/source/region/country/module/date/snippet
   ↓
DeepSeek 结构化筛选
   ↓
validate JSON
   ↓
dedupe by source_url/title
   ↓
render HTML page
   ↓
store in KV
   ↓
send Feishu summary card
```

### 7.3 DeepSeek 引擎边界

第一版只实现 DeepSeek，但代码保留模型引擎边界：

```js
async function analyzeWithLLM({ candidates, sources, period }) {
  return deepseekAnalyze({ candidates, sources, period });
}
```

未来替换模型时，只替换 `deepseekAnalyze` 的实现。

## 9. 错误处理

### 抓取失败

- 单个源失败不影响整体。
- 记录失败源数量。
- 页面底部显示“部分源抓取失败”。

### DeepSeek JSON 输出失败

- 第一次要求严格 JSON。
- 解析失败后，第二次请求 DeepSeek 只修复 JSON。
- 仍失败则推送错误卡片到飞书。

### 无高质量内容

飞书推送简短提醒：

```md
本周未发现高价值美妆法务动态。
已检查 69 个信息源。
完整页面：/report/latest
```

### 去重

- `source_url` 优先去重。
- 没 URL 的用 `title + source_name + country` 去重。
- 已推送内容 30 天内不重复。
- 同一法规不同阶段可再次推送：征求意见、正式发布、生效提醒。

## 10. 调度

每周一北京时间 08:00 推送。

Cloudflare Cron 使用 UTC：

```toml
[triggers]
crons = ["0 0 * * 1"]
```

## 11. 验证计划

### 本地验证

1. 解析 Excel 并生成 `sources.json`。
2. 检查 `sources.json` 每条源包含必需字段，且空分类已被规则补全或标为 `uncategorized`。
3. 抽取 5 个代表性网站抓取候选信息。
4. 验证候选信息包含标题、来源、链接；无链接的候选不得进入最终推送。
5. 调用 DeepSeek 生成 JSON。
6. 校验 JSON 可解析且字段完整。
7. 生成 HTML 页面并检查所有资讯均有文字超链接。
8. 生成飞书摘要卡片，确认只展示摘要和完整页面链接。

### 云端验证

1. `wrangler deploy`。
2. 手动触发 `/test`。
3. 查看 `/report/latest` 页面。
4. 检查飞书摘要卡片。
5. 检查 KV 中报告是否保存。
6. 检查重复运行是否不会重复推送旧内容。

## 12. 交付边界

本次优化只做：

- 信息源配置化。
- 官方/公开网页轻量抓取。
- DeepSeek 结构化筛选。
- 飞书摘要卡片。
- HTML 完整报告页面。
- 周一推送。
- KV 去重。

暂不做：

- 微信公众号全文爬取。
- 复杂登录站点抓取。
- 独立数据库。
- 用户后台管理界面。
- 多用户权限系统。

这些可以在后续版本单独设计。
