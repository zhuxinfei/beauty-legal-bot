# 美妆法务周报采集恢复、单卡推送与行动看板设计

## 目标

将当前“部分来源普通抓取 + 七段钉钉消息 + 放射状风险雷达”重做为一条可审计的周报链路：公开信息源失败后自动恢复，覆盖率不足时明确失败；最终只发送一条钉钉 Markdown 消息；图片改为移动端可读的管理层行动看板。

本次只修改测试分支 `codex/dingtalk-webhook-reliability`。真实测试通过且用户验收前，不修改 `master`，不切换正式客户 webhook。

## 成功标准

一次周报只有同时满足以下条件才标记成功：

1. 中国关键监管源覆盖率必须达到 100%。关键源指 `country/region` 为中国且 `priority` 为 `high` 的可抓取来源；直接入口失败时，可以由同一机构的官方备用入口恢复。
2. 全部可抓取来源覆盖率不低于 90%。覆盖表示该来源返回了可解析正文、候选链接或经过验证的公开替代入口，行业线索占位不计入覆盖。
3. 每个正式资讯条目都有可访问的 HTTP(S) 来源链接，并通过现有来源观察校验。
4. 六个业务模块均出现在报告中；无合格内容时明确显示“本周无高置信更新”。
5. 中国资讯在总览、图片和各模块内均优先于其他地区；同一区域内继续按风险、影响、相关性和置信度排序。
6. 新图片必须先发布到带报告日期的唯一 URL，并通过可读性检查后，钉钉才执行一次消息发送；Markdown 正文不超过 18,000 UTF-8 字节。
7. 行动看板在 1080 像素宽度和钉钉缩略预览下无乱码、裁切、重叠，正文等效字号不低于 28 像素。
8. 工作流、采集、AI 分析、图片生成和钉钉发送任一关键阶段失败时，进程以非零状态退出，不写入成功去重状态。

## 一、采集恢复

### 统一结果模型

每个来源返回结构化结果，而不是只返回候选数组：

- `status`: `ok`、`recovered` 或 `failed`
- `source_name`、`source_url`、`country`、`priority`
- `attempts`: 每次尝试的方式、HTTP 状态、耗时和错误
- `recovery_method`: `direct`、`retry`、`browser`、`alternate`
- `candidate_count`
- `final_error`

统计和覆盖率只依据该结构计算，不能再通过“是否生成行业线索占位”推断失败。

### 恢复顺序

1. 普通 HTTP 获取：完整浏览器兼容请求头、压缩、中文语言偏好和自动重定向。
2. 对网络错误、超时、429 和 5xx 执行最多三次尝试，采用指数退避和抖动。401、需要登录、验证码等错误不重试。
3. 对 403、公开页面空壳、依赖 JavaScript 渲染的页面，在 GitHub Actions Node 环境中使用 Playwright Chromium 进行一次公开页面渲染回退。
4. 浏览器仍失败时，按来源配置尝试官方 RSS、API、栏目页或公开备用 URL；备用入口必须属于同一监管机构或明确配置的可信来源。
5. 所有公开路径均失败后记录为 `failed`。不绕过登录、验证码、付费墙、IP 白名单或其他访问控制。

### 资源约束

- 普通抓取按域名限速，同域名同时最多一个请求，全局并发不超过四个。
- 浏览器回退按顺序执行，总数默认最多十二个，优先处理中国高优先级来源。
- 单个来源全部恢复步骤的总预算不超过 90 秒，整轮工作流超时调整为 45 分钟。
- 重试和回退结果写入日志，包含恢复成功数、最终失败数和覆盖率。

### 覆盖率门禁

覆盖率低于成功标准时：

- 不生成“正常周报已成功”的结果；
- 不写入资讯去重成功状态；
- 测试环境允许向测试群发送一条简短的“采集覆盖不足”告警，列出关键失败源；
- GitHub Actions 最终标记失败，便于追踪和人工处理。

## 二、内容质量与排序

AI 分析、来源观察校验和 30 天去重逻辑继续复用，但输入增加来源健康信息。

- 正式报告不得引用只作为线索、但没有可核验公开页面的内容。
- 总览突出中国监管变化、最高风险事项和本周必须执行的动作。
- 图片与正文使用同一份排序后的报告数据，避免“图片取前 10 条、正文有 16 条”的口径不一致。
- 图片中的总数是全部正式条目数；若展示 Top N，必须明确标注为 Top N，不能把 N 当作总数。

## 三、单条钉钉推送

### 消息结构

只构建一条 Markdown 消息：

1. 周报标题、周期和覆盖率摘要
2. 管理层行动看板图片
3. 中国重点和本周行动摘要
4. 六个固定业务模块
5. 数据口径、失败源数量和生成时间

每个模块内保留条目标题、地区/风险、核心判断、行动建议和来源链接。空模块仍保留模块标题。

### 字节预算

正文安全上限为 18,000 UTF-8 字节，使用以下确定性压缩顺序：

1. 高风险和中国条目保留完整核心判断、业务影响、行动和来源。
2. 其他条目压缩为标题、单句判断、单项行动和来源。
3. 删除重复标签、重复背景和低价值说明，不删除来源链接或模块标题。
4. 若仍超限，最低风险、最低相关性条目降为单行索引。
5. 最后仍超限时，移除最低评分条目并在总览明确写出“另有 N 条低优先级条目因单卡上限未展开”。

发送器保持现有网络重试和业务错误终止规则，但消息总数必须为一；不得自动恢复为七张卡。

## 四、管理层行动看板

### 版式

采用 1080 × 1440 竖版信息看板，替代当前放射状关系图：

1. 顶部：标题、周期、覆盖率和正式条目总数。
2. 第一屏：中国监管重点 Top 3，每项显示风险等级和一句结论。
3. 中部：六模块风险分布横向条形图，显示条目数和高风险数。
4. 下部：最高优先事项 Top 5，显示事项、行动、责任团队。
5. 底部：来源覆盖、失败源数和生成时间。

不再绘制所有节点都连接中心的装饰连线。颜色只辅助表达风险，必须同时显示文字标签。

### 字体与清晰度

- 使用可再分发的简体中文字体，并在渲染环境中显式注册或嵌入，不能依赖 Ubuntu 默认字体。
- SVG 为所有文本声明中文字体族和后备字体。
- 标题 48-56 像素，分区标题 32-36 像素，正文不低于 28 像素，辅助文字不低于 24 像素。
- 文本按实际宽度换行，卡片高度由行数决定；禁止通过缩小字体塞入长词。
- PNG 输出固定为 1080 × 1440 或等比例 2160 × 2880，不再通过高 density 生成 4000 像素宽但字号过小的图片。

### 图片发布顺序

当前工作流在钉钉发送完成后才把 PNG 写入远程 KV，消息发送瞬间可能读取上一期图片并被钉钉缓存。新流程必须按以下顺序执行：

1. 生成本期 SVG 和 PNG。
2. 将 PNG 写入带报告日期的远程 KV 键，例如 `asset:decision-map:2026-07-14.png`，同时更新 `latest` 键。
3. 使用带日期的公开 URL 请求图片，确认 HTTP 200、`Content-Type: image/png`、尺寸和最小字节数符合要求。
4. 将该唯一 URL 写入单条 Markdown 消息并发送。

任一发布或健康检查步骤失败时，不发送带旧图的正常周报，工作流标记失败。

## 五、测试与验收

### 自动测试

- 普通失败后重试成功、浏览器回退成功、备用入口成功和最终失败。
- 401/登录类错误不重试；429、5xx、网络错误可重试。
- 中国关键源和整体覆盖率门禁。
- 中国信息优先、六模块齐全、来源链接保留。
- 单条消息且不超过 18,000 字节；超长报告按规定压缩。
- 图片统计使用全部正式条目，Top N 标签准确。
- SVG 字体声明、画布边界、文本换行和 PNG 尺寸。
- 图片发布先于钉钉发送；消息使用带日期的唯一 URL，健康检查失败时不得发送。
- 业务失败不写去重状态，工作流退出码非零。

### 视觉验收

使用真实中文内容生成 PNG，并检查：

- 不出现十六进制方框、豆腐块或乱码；
- 1080 像素原图和约 720 像素钉钉预览均可读；
- 无裁切、重叠、超出画布和极小文字；
- 中国重点、最高风险和行动清单在首次浏览中可定位。

### 真实流程

本地测试全部通过后，将代码更新到测试分支，使用仓库现有 Secrets 手动运行一次 GitHub Actions。验收记录必须包含：

- 可抓取来源数、直接成功数、恢复成功数、最终失败数；
- 中国关键源覆盖率和整体覆盖率；
- AI 模块数和正式条目数；
- 钉钉消息数、字节数、重试次数和最终状态；
- 图片尺寸、字体检查和公开图片 URL。

真实流程只发送到用户已授权的测试 webhook。用户确认内容、版式、图片和覆盖率后，再单独处理正式客户 webhook 与 `master` 集成。

## 不在本次范围

- 钉钉知识库写入和文档 API。
- 绕过登录、验证码、付费墙或网站访问控制。
- 修改正式客户 webhook。
- 未经用户验收直接合并到 `master`。
